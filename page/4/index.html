Library notes from Malletts Creek

Out for a bike ride to Malletts Creek Branch Library after a delicious lunch of BBQ at Ricewood.

One of the challenges of low overhead custom postcard making is getting an image manipulation pipeline that kicks out photos that a appropriately cropped and sized. The latest challenge is identifying the single command line that will turn a photo from my phone’s camera into an appropriately sized image for the card with a minimum of cropping, distortion, and manual file transfers.

What I know doesn’t work is upload via Twitter, because the series of resizings generate visible compression artifacts.

I can upload from the phone to OneDrive, which gives me a photo of size 2592 1456 pixels (1.78:1, 16:9). The target size is 1875 x 1275 (1.47:1, 25:17), which is postcard format. I can keep all of the pixels in the image by letterboxing it, or I can crop off the edges. What I can’t really do is stretch the image to fit because it’s too far off.

Looking further at my camera software (Lumia Camera) it appears that I can also take photos at 4:3. There’s more experimenting to be done.

For more documentation, see Efficient Image Resizing With ImageMagick, Dave Newton in Smashing Magazine, June 25, 2015. Dave goes through a professionally thorough view of resizing and resampling images for the web, comparing his results against Photoshop. For instance:

I tested 31 different settings for -filter and got the best results with Triangle. The Triangle resampling filter is also known as bilinear interpolation, which I discussed above. It determines pixel color by looking at a support area of neighboring pixels and produces a weighted average of their colors. I found it best to specify this support area at two pixels using the -define filter:support=2 setting.

Dave uses DSSIM from https://pornel.net/dssim to test his results.

This tool computes (dis)similarity between two PNG images using (my approximation of) algorithms approximating human vision. Comparison is done in La*b color space using a multi-scale variant of the SSIM algorithm.

There are sites that have automatic crop algorithms, though I haven’t yet found a package that’s usable without licensing. The interactive web site of Croppola is one useful example of this genre.

The issue that 1.7.1 resolves for me is certificate support - if I understand the bug right, the Docker daemon was creating a key for each interface it had, but not all of the interfaces were up at key init time. The fix was to move DHCP init earlier in the start sequence so that everything was initialized by the time key creation took place.

The Docker team is trying to improve code quality by having their code base pass the golint test. The open issue, fix golint errors/warnings #14784, has attracted a swarm of contributors, and a lot of them have proposed changes that don’t pass the automated tests. The build farm is managed with Jenkins, and you can see the status of every build of Docker on their dashboard.

Having a full lintable codebase is a great goal. The challenge comes when the suggested changes by lint send you in the direction of making an incompatible change to some interface that has the “wrong” name but that people were relying on.

Perhaps importantly for the work I’m doing it’s worth noting that currently all of these tests are done on x86 platforms (Linux and Windows) and there isn’t an official automated build on ARM. I’ve been using the Docker distribution from Hypriot as my baseline for working on the Raspberry Pi, with good success.

The work of the morning is finishing up yesterday’s work on sending postcards with Lob.com. Key to this effort is resizing your favorite images so that they meet the template. Fortunately I had the presence of mind seven years ago to write up my notes on using sips and Imagemagick for image manipulation and that helped. In this case the magic incantation was

sips -z 1275 1875 original.jpg --out new.jpg

If you don’t have a Mac, or want to do this on a server or container somewhere, you’ll want Imagemagick. That venerable program has a bazillion options, and you’ll find the image geometry options at http://www.imagemagick.org/script/command-line-processing.php#geometry . Recreating this exact sips command is left as an exercise for the reader.

That bit of image processing taken care of, the next iteration is to launch a few actual postcards into the ether. I first ran into the “pocketmod” planning technique in 2005, and here it is 10 years later and there’s one in my pocket.

A pocketmod is a cleverly folded piece of paper with preprinted templates for things like a mini calendar, todo list, quadrille paper, checklist etc. It’s the smallest instance of a PDA that I have seen yet, requiring nothing more than a blank piece of paper and a pair of scissors to execute. You could even bypass the scissors if you had good paper and a crisp straightedge to do a rip.

Astoundingly, http://www.pocketmod.com/ is still online, in all of its 2005-2007 glory. That system does an 8-up PDF from a set of templates, so that you can print a lovingly detailed mini booklet. My own use has long since devolved into scrap paper plus rip down the center. When it fills up, invert the fold, and you have a brand new organizer.

19 July 2015

The weak ties of your social network are its greatest strength

Lior Zoref has a new book out called Mindsharing: the art of crowdsourcing everything (Portfolio / Penguin, 2015). I have a copy of it from the Ann Arbor District Library which was on the new shelf at Mallett’s Creek on a sunny and hot summer Sunday afternoon. Mallett’s has good air conditioning, a very good spot to be on a warm day.

The challenge of a book like this is picking out the good bits without repeating the author’s technique of using the word “Mindsharing” (capitalized) at every opportunity. I’ll try to unpack the approach with a simpler version of the strategy, which seems to boil down to this:

  • Make lots of connections online, even with people who you don’t know very well
  • Ask them really good questions when you are trying to make a decision

Lots of connections is the biggest suggestion here; rather than get help from a small circle of trusted friends, the crowdsourcing approach to human improvement relies on good ideas coming from large or very large sets of people (at least 250), and relies on pulling in unexpectedly good recommendations from people who don’t know you very well. Lior references Mark Granovetter’s “The Strength of Weak Ties” (1973) early on, but of course Granovetter didn’t know about Quora, Facebook, LinkedIn, and the host of other online properties and lists that allow for computer-mediated interaction with people at a distance.

There is an art to asking good questions that could fill a book. If you look on the biggest question-answering websites like Quora or Stack Exchange and start to follow along, you will quickly find that good questions get rewarded with positive feedback and bad questions (or impossible to answer questions) get ignored or criticized. Lior points out the there’s a very big difference between typing a query into a search engine and asking a live audience the same thing - the human element of the search allows a personal connection.

Lior is known for a TED talk (he brought an ox onstage) and is out promoting his book. The book itself feels like an ideal airplane read for someone who is aspiring to self-improvement, is facing a career change, and who hasn’t figured out what the point is quite yet with Facebook as a career tool.

Remembering Hazen Schumacher of Jazz Revisited

Hazen Schumacher was a radio broadcaster who hosted the popular “Jazz Revisited” show on NPR and Michigan Radio (WUOM) for 30 years. He died on July 18, 2015.

Arborwiki’s page on Hazen Schumacher has memories of him and his program. The jazz collection that he assembled and all of the master tapes for the show are now at the Jazz Museum Bix Eiben Hamburg, and 24 editions of the program are up online for your listening at http://bixrevisited.com .

Levenshtein distance for approximate match

The problem, simply stated, is that of typoes and inconsistent spelling and pluraliziation. Let’s say you have a collection of text that you want to annotate with some metadata, like categories and tags. You don’t want to set aside ahead of time exactly which tags you want to use, but you want to catch through some test or lint step that you’ve done something wrong by typing “apples” into a field that really has “apple” most of the time. A good algorithm will catch minor deviations from the norm and advise you when you should go back with a fresh pair of eyes and do an edit. The bigger problem is deduplication.

My first pass at a proposed solution was to look for a version of the old Unix program “uniq” that would have an approximate match. There is also an old Unix program “agrep” (“approximate grep”) from Udi Manber.

[WM92a] Wu S. and U. Manber, “Agrep - A Fast Approximate Pattern-Matching Tool,” Usenix Winter 1992 Technical Conference, San Francisco (January 1992), pp. 153-162.

Neither of these first lines of attack to see if someone had completely solved the problem before did the trick, but some searching around and a conversation with John Hritz hit the neuron that remembered the algorithm called the Levenshtein distance. That led me to Wikibook: Algorithm Implementation, and its set of Levenshtein distance code. It so happened that there was a very slow implementation in the bash shell included in that collection that was a cut and paste and two lines of code from being exactly what I needed. A little bit of glue code in awk and I was ready to go.

Now the problem with cut and paste two lines of code is that this particular implmentation is very slow, albeit correct. Given the choice between slow correct code and fast code that’s wrong (because of the bug that you don’t yet understand), I’ll take the slow code plus a comment and a writeup that describes how to make it faster.

What I’m really after is just the pairwise comparison of a set of words where the two words are close to each other. If the distance between two words is greater than some threshhold, I don’t need to go any further. That algorithmic shortcut isn’t in the library codes that I found.

The bigger question is how to write a checklist and implement it in code to make sure that a big body of work stays internally consistent. It’s one thing to have a categorization rule for your writers that says “always use the singular”, but very much something different to have Test 38 in your automated test corpus apply that rule and notice when it’s wrong.

Update 29 July 2015: Dennis Yurichev with a post Fuzzy string matching + simplest possible spellchecking + hunting for typos and misspellings in Wikipedia. shares some Python code to handle the case of automated checking against a probably correct dictionary. He uses this python-Levenshtein module to do the hard work.

Publishing from Hugo to Github Pages

One of the challenges in getting things started is figuring out where the end product is going to live. Hugo has its own web server which is pretty nice, but since it generates static pages, the results can really go anywhere.

This tutorial goes through the process in excruciating detail

http://gohugo.io/tutorials/github-pages-blog/

but somehow fails to make it easy. What I’d like is a publishing workflow that just like the normal Hugo workflow puts things online instantly once they are built.

This one deploys with CodeShip

http://jice.lavocat.name/blog/2015/hugo-deployment-via-codeship/

And this one deploys with Wercker

http://gohugo.io/tutorials/automated-deployments/

Yet one more, with Github hooks and S3/Cloudfront; it depends on s3cmd.

https://github.com/imperialwicket/imperialwicket.com

Finally, one that focuses on the github side of things:

https://gist.github.com/chrisjacob/825950

The approach I have runs like so, it assumes that you have a working directory “~/src/hugo” with your blog in it, and a target directory “~/src/vielmetti.github.io” set up for github pages.

cd ~/src/hugo
hugo -d ../vielmetti.github.io --theme=vacuum
# wait to build, then
cd ../vielmetti.github.io
git add -A .
git commit -m "build complete"
git push origin master

By having separate build directories like this you can keep the continuous build on your draft setup going, and then push a working version whenever you’re perfectly happy with the result. Not like anyone is really perfectly happy with any blog.
