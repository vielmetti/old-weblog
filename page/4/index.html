5 October 2014

using "jq" for command line web applications for civic data

One of the tools that I rediscovered and have been really happy for having done so is "jq", a command line tool that bills itself as "awk for json".

I've been writing awk code since 1985, and some limited subset of it is something that I know really, really well. awk, however, is from the punchcard era, and as such it likes to deal with records that are all on one line each. Parsing JSON in awk is very clumsy and ad hoc and really doesn't work all that well. Since most Internet APIs these days have some kind of JSON encoding, it means that you can't simply dash off an awk one-liner to consume and transform Internet input data.

jq fixes that situation. Here for example is a one line renderings of a common task I look to solve with municipal data as a test of any new tool development: analysis of parking data. This code sample prints the total number of open spaces in the Ann Arbor Downtown Development Authority's garages:

curl -s http://www.a2dda.org/map/AADDACount.json | jq "[.countdata[].spacesavail | tonumber] | add"

I'm not going to try to explain how jq works, except to note that it's constructed in order to be a filter: data comes in one JSON object at a time, and the script iterates over them, transforming them in some way and then passing JSON out the other end. This makes it perfect for ad hoc Unix pipeline efforts where you're chipping away at a data set trying to make sense of it by successively refining it as you go.

I've built jq pipelines for this Ann Arbor parking data, for AAATA bus system data, and for the Marquette Park Cemetery data. The parking data is straightforward, since it exits the system as JSON. Bus system data is only incrementally harder; it started either as HTML or as XML, and for that you want to use "tidy" and "xml2json" in your pipeline. The Park Cemetery data set started as JSON via the Socrata SODA API, but I needed little bits of "sed" and "perl" to smooth out some rough edges in the source data; it really wants to end up as GeoJSON when I'm done, but it isn't there quite yet.

Shell programming is my favorite programming environment. Any time I can take a set of well-understood tools and crunch through big data sets with only a few lines of code, I'm happy. The challenge of shell programming is that it's full of opportunities to get things just a little bit wrong with parsing; by making JSON the data format that's shoveled between programs rather than a flat one-record-per-line text format, you open up the opportunity for pulling apart rather complex structures and manipulating them with hardly any work.

For further reading, see Jeroen Janssens new book Data Science at the Command Line which works you way through dozens of tools like this that can help manage big data without big programs.

Notes on building the Marquette Park Cemetery finder app for Code Michigan

I'm working on a tool or a set of tools to assist people in finding grave sites in the Park Cemetery in Marquette, Michigan as a part of a project for Code Michigan. I'm working with a team that's based in the U.P. to do this, and we're collaborating over Google Hangouts. Code Michigan is a hackathon, so this project timeline is compressed to a single weekend. The team is Edward Vielmetti, Jim Argeropoulos, Chris Marr, and Lynn Czarnecki Makela.

My motivation for picking the grave finder app is a personal one. My grandmother, Ada Burt Vielmetti, is buried in Park Cemetery, and I've never visited the grave because I didn't know where it was. The cemetery covers over 100 acres so it's impractical to just wander around until you find what you're looking for, and there wasn't an online finder's aid. When the Code Michigan challenge came about it was an obvious task to tackle.

It turns out that providing detailed location data for individuals seeking to find people inside this cemetery is a common civic task. The role of the sexton, aside from care for the physical properties inside the cemetery, is to guide people to where they are looking for. This could be the grand burial grounds of famous and wealthy people with a beautiful mausoleum, or it could be a simple marker underfoot. In any case the challenge is to provide someone with enough guidance so that they can get close enough to the spot that they can locate exactly the one spot that is of interest.

Some notes on technology:

The maps problem for any geolocation task relies heavily on work that's been done before. Base maps of the city are available from Google, but the Google panopticar has not traversed all of the cemetery lanes and the Google map doesn't show the structure of the roads that you can walk or drive down. Open Street Map is better, in that it does have more detail within the cemetery, but it doesn't go down to the individual grave level.

On top of the base map with roads, there's a plat map showing the sections of the cemetery that's available in PDF format from the city. Our team took this plat map, field checked it, and encoded plat boundaries as GeoJSON files so that we had an accurate representation of the area within the cemetery that an individual could be located. This brings the task down to a range, rather than a specific point, but it's practical enough and a small and precise enough area to be a useful first step.

The data about individual burial locations in Park Cemetery was provided to use using the Socrata SODA API. A challenge here was that the "location" field, a critical one for our use, was encoded in such a way that a simple download of the data would not yield an immediately usable result since the location (in latitude and longitude) of an individual grave is not known. We addressed that issue by using the API's JSON download function to return not only the valid fields in the data but also the ones marked invalid, and a post-processing step cleaned the data to extract a valid "plat" field from the provided data.

JSON is a very easy to manipulate format, and I used two tools that are documented in the new book Data Science at the Command Line by Jeroen Jansens. "jq" is like sed for JSON data – you can use it to slice and filter and map and transform structured data with the same ease that sed, awk, grep and friends let you play with text. "json" offers powerful and flexible tools for taking JSON data and printing it in flexible formats. Between those two tools and a bit of other Unix command line efforts with "sed" and "perl", it was possible to write a handful of one-line transformations that take the Socrata data and turn it into a more compact JSON and HTML format for use directly inside applications.

By running these transformations on the 12,000 plus record original dataset, I was able to pull out just the location information and basic personal details for everyone - including my Nana. This gets turned into a series of files, one per cemetery section, sorted by lot number so that there's a chance that you could walk a section methodically and have the data be your guide to whether you are close to what you're looking for.

To publish the data in a format that is useful to the world, I turned to an old favorite, Localwiki. This comes with a number of useful tools right off the bat. At one file per cemetery section (e.g. Jenney W the sorted HTML list loads easily into the list. The GeoJSON we created can be imported using the Localwiki API, or cemetery areas or points can be hand-plotted on an OpenStreetMap based map. Names for which Localwiki has more information can be linked. (I resisted the temptation to link every name, though if you had more coverage it would be the obvious next step).

Localwiki is good for this effort for a few reasons. It has a local search tool that lets you type in a name and get something in return. While not as precise as a purpose-built search tool, it's easy to implement, and at 12,000 records spread out into a few dozen sections no individual file is too big to load. We know that Localwiki is good for local search, and eventually when the pages are indexed they will be reachable by Google search as well.

The community aspect of civic data use with Localwiki is an important one. No municipal employee can ever hope to gather biographical data on 12,000 cemetery residents, but we want to support citizens in their interest to collect and refine as much detail as they need to memorialize or to research personal and community history.

In addition to loading Park Cemetery data into Localwiki, we also pursued a prototype of a purpose-built application designed specifically for mobile phone sized screens that would provide a name index and a plat finder for people who are on site. The goal is to reduce the complexity of the problem down to type in a name and get something back on a map that includes a dot saying where you are. This system uses Drupal as a back end, with maps provided through a responsive front end system specially adapted to look good on a mobile phone sized screen. Links are provided to Localwiki for reference data, and the design includes ways for individuals to quickly add to or verify location information so that you can pinpoint a spot.

Some lessons learned from the effort, which isn't done yet but getting close!

It's important to have a variety of skills on your team. We divvied up the problem into parts that were simple enough for each of us to tackle based on our own skills, and sorted through coordination issues via periodic Google Hangout calls. When you're working using tools you are good at and on things you find challenging but not impossible, there's a lot of room for making progress in a short time.

Tools and standards help. We discovered GeoJSON mid-day through day 2, and were able to use it almost immediately to extend our data to be reusable without recoding on multiple platforms. Similarly, the combination of the Socrata SODA API and the powerful "jq" and "json" tools turned a complete data processing problem into a handful of lines of code.

When there's a real problem to be solved, it's easier to see that you are successful. Two members of the team have family members buried in Park Cemetery. With the toolset we developed, we were able to find those graves, photograph the headstones, and decide how to incorporate that data into Localwiki. Knowing now that the problem is solvable with the data provided, it simply follows to extend that solution to a broader set of tasks.

Usable prototypes are more valuable than pixel-perfect finished products. Localwiki let me test ideas almost immediately, without much more than a little bit of coding needed to get to a point where a search engine would return a useful result. The tooling we built for the mobile app got to a usable demo phase promptly. A handful of photographs from the field were enough to not only field-check the data that we had, but also to better understand what we could do when there was more time to do it.

A weekend is not a lot of time to get much beyond the prototype stage.

We're not done yet. The contest rules want a video, and we're working on the storyboard. The coding is done, but Localwiki is never really done, and there are some pages to make more spiffy and full of useful content. But in some sense I feel like we've already succeeded, in that a completely hidden system got turned into something that I'm starting to understand.

The proof, perhaps, is that we were able to put together a team that answered the question: where is Nana buried? The proof below.

Some notes on open data for

Throwing out some notes here for a panel I'm on at the #micities conference at the Michigan Union today October 4, 2014.

The questions are around open data, considered as data published by a government agency that has a

  • well defined data format (CSV, Excel, JSON, XML documented API, etc)
  • well defined access path via Internet (HTTP, FTP, API)
  • well defined terms of use and reuse

Census data is perhaps the most openly open of open data. The worst would be PDFs of spreadsheets only available via FOIA, which fail on data format, access, and copyright.

There are multiple audiences for open data, each with separate goals and plans for their use of the data. A lot has to do with risk taking in the use of the data and in where the benefits to use flow.

  • enterprise: well understood data, no question about long term access
  • startup: questions about access, format, permissions; risk taking, experimental, commercial
  • civic: using data to guide policy or management of resources; not overtly commercial
  • makers: experimental, interest driven, risky but not necessary political or commercial

I don't see that breaking down use by status (private, public, non-profit) is the right way to slice this. You can have non-profits with enterprise-class IT operations, only willing to try things if they are sure they will work, or non-profits with goals that allow them to experiment. The focus on willingness to cope with uncertainty is the division that seems to be most helpful.

The Value of open data initiatives depends on the audience. Not every data provider recognizes or acknowledges every possible audience. Some data providers and gatekeepers are actively at risk if the data under their stewardship is opened up too much, so there may be a negative value to data access.

Some metrics for usefulness include

  • profit or cost avoidance: simple easy to reckon monetary measures
  • opportunity seeking, potential value, chance for risky ventures to succeed
  • good governance, improve (or at least change) civic involvement in decision making
  • civic engagement, getting people involved with operations as a goal unto itself

Controls for civic data are in part administrative and in part political. State and federal laws constrain what kinds of data can and should be released, for privacy, security, and policy reasons. Beyond that, there are costs associated with open publication that have to be weighed against budgets. Sometimes the publication of data imposes costs on an agency that are not recovered, and the profits or gains flow to private groups.

the internet of things world hasn't condensed into a single system, and because of its origins it's unlikely to do so.

The internet of things is a multi-protocol internet, with lots of legacy protocols in use as well as homemade protocols over TCP/IP and also serial port stuff and funky radio uses. Oh, and HTTP too.

Some of the things in the internet of things are very small and have very low cost network capacity, which rules out using a full stack HTTP implementation. Others are not quite as small, but still don't have enough oomph to take on an Internet's worth of traffic, so you might run HTTP but still keep the devices firewalled very carefully from the Internet.

If I were looking at standards and formats I would look at traditional Internet protocols and formats (HTTP, JSON), at older serial data networks (like RS-232 or even MODBUS), and I'd get a grasp on how you might communicate over radio networks like the 915 Mhz ISM band or Bluetooth Low Energy (BLE). I'd also look at publish/subscribe systems like MQTT, and lightweight data retrieval systems like COAP, and I'd be aware of all of the things you might do over a simple TCP connection. And then I'd look at the security of all of this (and weep, sometimes, because it's not easy.)

The internet of things world hasn't condensed into a single system, and because of its origins it's unlikely to do so. Picking which of three or four possible solutions could work for your application - one of which might be HTTP - is good. Deciding which fits the right sweet spot of skills, costs, and time to deploy is another good way to think about it.

The Bee App, a multiplatform issue tracker client for Mac

I've been tracking my personal todo list on Github for nearly two years now. When I first wrote up the experience I had just been using the standard web client for the system; since then I've added Octodroid as an Android client. So far so good, I've closed more than 2300 issues since I started. Github's issues is a completely feature-filled issue management system, far and above overkill for a personal todo list, but useful and complete enough to manage routine tasking.

The new thing in my life (to go with the new laptop) is Bee (@thebeeapp), a Mac native client that's a front end for several issue tracking systems. I have a free 30 day trial, after which the software looks like it could be reasonably priced if I were to use it all the time. Notably it's software-as-software, not software-as-a-service, so I'd just buy it and not have to pay every month for it.

Bee gives you a nice GUI for your issues, so that you can sift through the debris of what you thought you wanted to do and comment on, reassign, close, or edit those things. The net effect for me over a few hours of use was to flush out about two dozen things that had been obscured by my previous issue management regime that I was never going to do or that were totally overtaken by events.

The "getting things done" ritual speaks of a weekly review, where you once a week sort through all of the detritus that had landed on your todo list over the course of a week and make sense of it all. The ability to take on a review like that depends crucially on whether you can get everything that's on your plate into a single list that you can walk through methodically (and in some sensible order) so that you're not horribly fatigued by issue overload by the end. Perhaps it's the new app smell that makes it possible for me to do that anew, but I was really happy with Bee as a way to work through a review, and I'm hopeful that it keeps giving me insights and inspiration as time goes on.

Accumulating editorial debt and dealing with out-of-date text

Editorial debt accumulates when the passage of time makes a document less true, to the point where the written word has to be edited to keep it up to date or scrapped because the work is uselessly outdated.

For fast-moving stories, the first edit will be wrong in many ways. There will be missing facts, incomplete images, and gaps in the narrative. This is short term editorial debt, which can be addressed once the immediate crisis has played itself out. A proper self-contained story will be true in its time, even if the story so far is incomplete.

For slow-moving stories that you want to keep accurate over time, you write things that pretend to be timeless but that are no longer true. Details too small to warrant a complete rewrite of the narrative are now wrong, and the text needs minor but significant edits to reflect current reality.

A similar concept is bit-rot, the notion that software degrades over time and no longer works as well as when it was released. That can occur as a result of changes in the operating system, changes in the working environment, adaptations by users and appropriation of features that were not completely thought through, or just changes in fashion.

How to mark things that are signs of editorial debt to come? Words like "now" are dangerous, or "currently"; the page does not know when that time is past. It would be better to explicitly date the elements that are tied to time, so that you can instead say "in October 2014", which will always be true even if circumstances change later.

What results can be a patchwork of text that tries to be universal but that always looks dated. Balancing the narrative with "it used to be this way but now it's this way" versus deleting the old obsolete text and just starting fresh, now there's a challenge.

Wiki is really bad for accumulating editorial debt. Text written in an encyclopedic style really wants to be true, and not just a snapshot in time of one of many possible states along the evolution of an idea. Consider a timeline as an editorial structure that puts the age of the idea primary in the narrative, so that it's clear when the idea that you had needs to be revisited.

Compare with "technical debt", used to describe code that needs to be cleaned up that drags down progress.

Do you still write 750 words a day?

I'm sure that I write way more than that in aggregate, but I've not been keeping up with what was once a habit I attempted: to write a 750 word essay every day.

I know that it's possible to sit down with a single cup of coffee and a single idea and no distractions and crank out 500 words at a sitting. It helps if the coffee doesn't spill, and if your todo list is in check, and if you have a full sized keyboard and not a tiny phone. The morning ritual of daily coffee and wifi usually produces something publishable, though often it's much shorter. You might only clip and save and share a single tweet's worth of someone else's work and call that a success to capture the idea of the moment.

I don't sit down and write all the time, though. There's meetings to go to, code to struggle with, people to see, places to go, dinner to cook, and the endless slog of Internet to wade through. When I do have a chance to write to produce things, it's often 50 words there, 10 words here, a click of a button there. It's not always long, extended efforts.

The long, extended efforts are fun, don't get me wrong, and they really focus the mind and help me think about topics for much longer than I would otherwise. The ideal writing environment gets all of the distractions out of the way and shuts out the beep-beep-beep of interrupts that otherwise drive my efforts. Getting away from interrupts turns out to be really hard work, and obtaining an uninterrupted stretch that allows such a focus is a rare treat.

What would you focus on if you had 750 words of uninterrupted focus every day? It should be enough to dig deep into a topic, not just to hit the surface but to wallow around in it, trying out lots of options.

I'm cutting this self-reflective essay off at about 300 words, because there are 450 words that I'd rather apply to something else.

A design for heat maps of parking system congestion

The unit of analysis is the parking structure. It has a capacity c, and a usage u(t) for any time t observed.

If u/c > 0.8 the structure is congested; at u/c > 0.9 the structure is full. That reduces the time series of usage for a given structure into one of three values (green, yellow, red). Some public policy goal wants to keep the structures congested (to maximize revenues) but not full (to minimize complaints).

Given a series of values over a range of time, you want to summarize those numbers. Do you pick the worst value ("we were temporarily full"), the average value, the mean value? Some decision needs to be made. If you have frequent measurements you could conceivably show this as a continuous series with lots of data points.

Heatmaps in R gives you one way that you might visualize a data set, if it were properly constructed and sampled as needed. Creating Heat Maps from Excel Pivot Tables with Conditional Formatting does the same treatment in Excel.

keeping up with the world, and the world keeping up with you

How do you organize materials such that they don't get lost, so that they stay reasonably up to date, and so they work the way your mind expects them to? Wiki and weblogs have been one of my answers, but even with that it's hard to wiki and weblog hard enough to keep up with the world.

I'm noticing this especially with the contacts database on my computer. With nominally thousands of names there, I'd have to update more than a dozen names a day simply to say that I was current within the span of a year. It's simply not going to happen! Ditto in spades the wiki-conundrum, the more you have in your wiki the more likely it is to be outdated.

Perhaps the answer is to invert the problem. What do you do to make the world keep up with you? Is it easy for someone to see what you want them to see, or is your external presentation confusing? I'll note that it's easy for people to determine that I have some interest in power outages, though that's hardly the focus of my life these days.

that new computer smell

I won't be complaining about a small keyboard or a small screen for a while, but I have a lot of tools to reinstall on a new system.

In order of appearance:

Chrome, first thing. I like Safari enough in a pinch, but when it comes time to sync up my working environment, Chrome is what I need. A bonus on this system is that there's a working full-screen mode.

git comes with the machine, so I didn't have to install it; it's a big part of making sure that stuff gets version tracked.

Node-RED. I need to have a simple programming environment that is network and device aware, and for the past few months Node-RED has been it. I'm at version 0.9.0 now. That depends first on having node.js installed, and subsequently on pulling in a lot of Node-RED dependencies.

Contacts. I've synced Google Contacts with the Apple address book; that took an extra step since I'm using 2-factor authentication at Google. The CardDAV support is new to me.

hearts, spades, clubs, diamonds, and bots

Pete Kaminski pointed me to this essay on archetypes in computer gaming, in which four types of play style are described. Hearts play for social interaction, spades play to explore, diamonds play to level up, and clubs play to create havoc. It is a nice description of state of the art online game play in the MUD and MOO era of online gaming.

What has changed a lot since those olden days is the emergence of a new class of player, one who builds and operates bots to amplify their own game play. It is not enough to win friends by your personal charm in this world; instead you write the Bayesian classifier that guides your behavior scientifically towards friend maximization. Whatever can be automated will be automated, and the rest of human behavior analyzed to give clues for further automation.

Lots of online games lend themselves to machines assisting or even taking over. People who have substantial bot writing skills have an undue influence on the culture of online networks and games for the few humans who stick around unarmed to watch the fray up close. What better way to while away the hours online than by tending to your collection of bots that prompt, provoke, or guide the behavior of other humans (or other bots).

28 September 2014

the late age of blogging

The revolution will not be blogged. It will be twittered or facebooked or elloed or plurked, but blogging is not a revolutionary activity in the new era of the net.

The process of writing for your own web site and then announcing that you have something new to say to the world is perhaps the least efficient way of getting news to the world. It is so much easier and more efficient to drop that news inside a walled garden of publishing and have it ricochet quickly inside the walls.

If blogging is old school, and slow and cumbersome, then why bother? I can only point to the benefits of having at least the illusion of control over your own words. The instant message instantly delivered inside the AOL chat room or Facebook page or you name your walled garden of choice gets to where it's going fast, but once it's off the screen it's gone. Blogging at least gives you the hope that if you wanted to see something from 10 years ago that it would still be there.

Enduring is a virtue that lends itself to systems that sometimes are more cumbersome for their long use. I am reasonably sure that I will be able to locate this post in 5 or 15 years, even if the nature of publishing on blogs means only a few people will read it the same day it was written.

do next?

One algorithm for deciding what to do next is simply to identify the item with the closest deadline and work on that. There is nothing like a deadline to focus the mind, and it is better to get the items that demand short term attention into the calendar and to allot the time to tackle them. Bernie Galler described this approach to me once, and it stuck.

A second approach is to put everything you might ever want to do in a big pile and pick something at random from it to work on. I use github issues to track my big pile of things to do, and alas it does not have a random() function. Fortunately though there is a Github API, and that combined with jq and my randomline script are enough to approximate that.

A third very reasonable approach is to ask someone what they would like you to do; or even better yet anticipate what they would ask and do it before they ask.

For machines, by machines: Wikipedia is for bots

What is Wikipedia for? It's for bots.

This machine kills trolls: How Wikipedia’s robots and cyborgs snuff out vandalism writes up the work of the Wikipedia contributors who have written robots (bots) that automatically extirpate contributions to the encyclopedia that are unwanted.

Alas, the result of all of this highly efficient robot-driven editing is that many new editors first interaction with the system is not with a human being, but with a faceless robot. The real worry is that the crowd of new editors that used to flock to the site to provide modest and well-intentioned edits are being systematically pushed away by the overroboticism of the systems administration.

Bot vs bot is a long term war that has no human winners, and the quality of the resulting system is poorer because the bots feed largely on newbies. Only when you master the gauntlet of robot overlords do you eventually get welcomed into the inner sanctum of humanity. Those who stay will be always watched over by the machines, and only a few in the Wikipedia elite will master the necessary skills to automatically patrol the recent changes based on their programming efforts.

Inspired by a fascinating talk at ICOS by Wikimedia Foundation researcher Aaron Halfaker, who might come to different conclusions from the same data.

26 September 2014

write every day

How do you produce so much writing? It is simple. All you have to do is sit down every single day and write.

There is nothing like a daily deadline to focus the mind.

Weblogs do not impose an external deadline. No editor is asking you when your work is going to be ready, and no producer is going to feed you copy. It is just you up against the elements, bravely typing away alone and hoping that your focus is not distracted by systems problems.

Writing all the time is a habit. Some parts of the writing week are easy, when it is just you talking to the page and getting fully formed thoughts out as fast as the keyboard or screen will let you. Other days it's a struggle just to come up with anything that anyone would possibly want to read again.

The weblog is a mature writing format. The tools are not evolving, and new startups are not appearing with whizzy new high concept takes on the form. The fancy new school tools are for essayists (Medium) or for micro bloggers (Ello). I am happy to see incremental improvements at Typepad to make responsive designs easier to put together, but that is a refinement and not an evolutionary change.

Due to an unfortunate incident with a cup of coffee, my medium of choice at the moment is a mobile phone browser. The small screen gives me about a paragraph of text as context for my writing, and the input box on Typepad's dashboard is just competent enough to make this no harder than it should be.

I think about other things I wish I did every day, knowing full well that there are only so many daily habits you can sustain. Facebook wants to catch you every day on the birthday treadmill, your dentist wants you to floss your teeth, the pedometer wants to be logged, the dishes washed, the cats fed u.s.w. My morning wants a cup of coffee, and the daily coffee and WiFi lends itself to blogging.

I aspire to being a daily blogger, but I am not one. It is good to have goals.

A design for an an Ann Arbor events channel, @a2events

Here's a design for a low overhead way to post about events coming up in Ann Arbor. Follow @a2events on Twitter if you want to play along.

The first rule of business is that every event has to have a URL, and the more simple and direct the URL is the more likely it is that it's authoritative for the event. If an event doesn't have a URL yet, I'll blog about it here if I need to post about it.

The second rule of business is that I'm not keeping a calendar, just a feed, and the feed is not going to be a 100% comprehensive feed; you have to be prepared to get events info from multiple sources and I'm just one of them.

The third rule of business of course is that if you want to pay for this service we'll find a way to let you pay, though the specifics are TBD.

My basic idea for technology is to use a tag on my Pinboard account to feed a specific tag, and to use that tag to feed IFTTT to feed Twitter so that events get pushed out. I'll probably also use a Node-RED page to help construct something that does the right thing, and there will be a Slack channel on the a2b3 slack to do - something.

25 years of professionals eating lunch together on a regular basis

I wrote up a piece about how you might rewrite your resume to provide a "retcon" ("retroactive continuity") treatment, which emphasizes reinterpreting past events in such a way that they look like part of a story with a narrative arc.

That got me to thinking about lunch, of course, and the little hoary old story I tell about when the a2b3 group started "8 years ago". And I realize now that I can trace back continuity way longer than that.

Once upon a time, when Sun 3/50s roamed the earth, I worked at OTA Limited Partnership as a systems guy. The company traded options, and I was a junior part of the ops team that kept the trading systems running. I wrote some code, I did some sysadmin work, I debugged flaky Proteon routers, I made sure we had a Usenet newsfeed. Life was pretty good.

The market was open every hour that lunch was served, so it didn't really ever make sense to leave the building during trading hours. That mean that lunch was brought in for everyone every single day. We had an elaborate scoring system where instead of you always buying lunch for yourself, you'd buy lunch for the team, and you'd get tally marks on the board to make sure that you properly were credited plus or minus depending on the number of lunches you bought. The bookshelves had a big thick binder full of menus, and we had some favorites.

That was my first real experience with professionals eating lunch together on a routine basis, and I'm still friends with the OTA crew, long after the last tally mark was erased. That dates to 1989 or so, which means I've been eating lunch as a grownup for about 25 years.

I realize now that if I were ever to get a real job again, one of my interview questions would have to be "what do you do for lunch around here". If the answer is wrong, I can't take the job.

Thanks to David Bloom for prodding me to tell more of a back story, and to my former daily lunch companions at OTA (including specifically John Hritz who is an a2b3 regular) for initiation into a professional lunch ritual.
