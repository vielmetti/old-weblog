How to squash a commit in git

If you are working on a project in Github, your work isn’t always linear. Sometimes the change you make doesn’t work completely, so you back a little bit of it out and move forward on a next part. It may take some hacking and sawing (and testing of course) to get a change that just works right.

Projects like to have contributions, but project maintainers don’t like to insert noise into their efforts. The ideal commit to a project is completely clean: one single commit that incorporates all of the changes that you want to do, without any of the intervening cruft.

Thanks to Alan Gutierrez for the help here.

In the lingo, this is known as “squashing a commit”. It turns A->B->C->D->X->Y->Z into A->Z. Here’s how you do it.

First, set up your environment before you start to work. Create a branch to put your changes in and work within that branch. You can do all of that from within Github, or from the command line as

% git checkout -b try/branch
Switched to a new branch 'try/branch'

When you’re ready to merge, create a pull request for that branch. That will gather together all of the changes you have made and drop them onto a single page for inspection. Look at the first change that you made in that sequence, and find the parent node for your first change; we’ll rebase off of that parent. This was the crucial bit that I was unable to figure out from the documentation.

Squash all of those commits for the branch into a single commit. There is no git squash command, alas; but there is git rebase -i.

% git rebase -i <after-this-commit>

Next, apply this single squashed commit to your master branch.

Further reading:

  • From Git Ready: Squashing commits with rebase
  • Makandra Cards: Squash several Git commits into a single commit
  • David Walsh: Squash commits with Git
  • Pro Git, Scott Chacon and Ben Straub: Git Tools - Rewriting History

Snap CI

Snap CI is a cloud-based continuous integration system. Like Travis CI, which I have written about here, it lives in the cloud, lets you run programs triggered by GitHub commits, and offers a testing environment for free for open source projects which lets you familiarize yourself with the system before deciding to pay for it for your non-open-source efforts.

I’m looking at all of these tools now because this weblog is now managed through a process that involves checking things into Github, which makes it completely suitable for automation for common tasks every time I save a file. (Spell checking, link checking, really any kind of quality control you can imagine reducing to a tiny shell script.)

Like Travis CI, Snap CI lets you automate the build process. Where it diverges (and where it’s interesting) is that the build process is only the first part of a multi-stage pipeline that includes deployment at the end of the task only when a set of human decisions have been made as to whether the thing you made is ready to be shared.

This Snap CI blog post on why Travis CI and Snap CI are not the same thing is from Snap CI and it’s a useful overview.

The downside of multi-stage pipelines for deployment is that they are very hard to automate correctly, and very hard to demonstrate correct behavior with open source projects that generally struggle to have any automated test facilities at all. So while Travis CI is a very good fit for open source components (and has attracted a whole sub-industry of writing .travis.yml files to support those), it really doesn’t have the whole “continuous deployment” pattern sorted out. And while Snap CI looks like it would be great for saying “once we’re happy with this, push it out to the world”, that setup is not easy to pick up simply by inspecting other projects that have done it.

Into the matrix with Travis CI

Travis CI is a continuous integration tool, a description which means nothing to people who have never used it. “Continuous integration” is one of these industry phrases that doesn’t serve to illuminate the problem that it describes.

Why would you care? Well, you’re working on a project with a bunch of other people, and it’s kind of complicated. It might depend on some old code that’s fragile and should be tested all the time when it changes, or it might depend some brand-spanking new code that doesn’t quite work and you know it doesn’t work yet but you want the test system to be less tedious.

What else looks like this?

  • CircleCI
  • Snap CI by ThoughtWorks
  • Solano
  • Appveyor (for Windows)
  • Jenkins (self hosted)

What does this look like from a workflow perspective? Every time you check in a file, the whole system builds, and all of the tests you have configured run. With proper configuration you can build a matrix of versions or configurations to test, so instead of a test running on a single system it can run on 3x3 or 2x4 or even 3x5x2 configurations.

Ruby

Travis CI’s default langauge is Ruby, and a lot of its tooling is written in Ruby, so its Ruby support is quite good.

Need a bunch of versions of Ruby? Use rvm.

  • Building a Ruby Project, Travis CI

C and C++

The version of gcc and g++ on the default systems is 4.6, and you can pull in 4.8 through ‘apt’, but you currently have to pick up 4.8 through the ubuntu-toolchain-r-test source and not simply by picking a gcc version.

  • rla/fast-feed/.travis.yml

Go (golang)

Go (golang) has not gone through a lot of versions yet, and its latest 1.5 build is written in Go. Because the language is so young, there is not lots of old code to port, and thus not too many version dependencies in libraries to worry about yet.

Testing two versions of Go (golang) plus Linux and Mac is easy, and the configuration file doesn’t have to do anything funky.

  • miekg/mmark/.travis.yml

NodeJS

Node 4.0 is new, and the source currently of a fair bit of porting efforts as a lot of things have broken not in the language but in its libraries. Fortunately, the nodejs and iojs split has been reconciled, but unfortunately there’s about six months of reunification work to be done.

If you are porting or building to NodeJS 4.0, you have to drag in gcc 4.8. Previous versions of gcc/g++ aren’t capable of coping with the C++11 constructs that are in V8 and (especially) in nan.

NodeJS 4.0 on Mac? In addition to gcc 4.8, you have to drag in nvm. The approachable way is to pull in the source from Github that lives at https://github.com/creationix/nvm.git, build it, and then use the thing that you just built to subsequently pull in the right version of Node.

  • rla/fast-feed/.travis.yml
  • sass/node-sass/.travis.yml

Python

Python 2 and Python 3 are both available on Travis CI, including even the latest Python 3.5.

If you need three versions of Python plus three versions of Django: use ‘env’ to drive ‘pip’, and ‘matrix’ to skip some builds where you know that the combinations don’t work.

  • (datadesk/django-softhyphen)

Lots and lots of Python versions, with different dependencies for each version? The tornado package has that.

  • (tornadoweb/tornado)

Looking to test Django plus Postgres?

  • (http://blog.schwuk.com/2014/06/13/using-travis-ci-for-testing-django-projects/)

Java

Java is supported.

If you need Java 8 in a particular version, see the below issue on Github Issues.

  • Java 8 version update and bug, Github issues.

Fortran

If you’re building scientific codes, Fortran is available through the apt mechanism.

Prolog

Prolog? Bring it in with apt-add-repository and apt-get. This is illustrative of Travis CI’s ability to incorporate nearly any language, since you can install new compilers and interpreters via package systems and then run them.

  • (rla/simple-template)

Multiple langauges

Two versions of Go plus Python plus an external dependency that’s being flaky? Use language: python because it’s easier to install go from package managers than it is to drag in the whole complex python multi-version infrastructure from scratch.

  • (miekg/mmark, in progress)

OS X

Use ‘osx_image’ to tweak the version of ‘osx’ that you get.

  • (http://docs.travis-ci.com/user/osx-ci-environment/)
  • (http://blog.travis-ci.com/2015-09-09-xcode7-gm/)

Both Linux and Mac, in general

Go is easy. (miekg/mmark)

Node requires bringing in nvm and gcc 4.8. (rla/fastfeed)

Python … (don’t know)

Prolog … (don’t know)

Ruby … (should work pretty well, but not tested)

Old code and old tools

One of the problems with old codebases is that they are missing modern tooling for automated builds, automated testing, and the like.

A second problem with old code is that the maintainers (absent modern tooling) may not know or recognize what dependencies they have.

If an old package is not mostly self-contained, it’s likely to have dependencies on other old code. This makes it hard to build, as those older dependencies have either gone offline or have mutated so that the old package doesn’t work with the new releases.

To confound the matter even more, the newer tools for automated builds, automated tests and so forth have their own sets of complex dependencies, so much so that unless you are around some team that has already absorbed these systems that you are unlikely to grasp all of their deeply interconnected complexities.

It’s easy to cascade into a deep chasm of interlocking dependencies, yielding enough complexity in the build process to frustrated people who really care more about the code than how it’s packaged for distribution.

I’ve been working this weekend on tidy-html5. It has a Makefile that’s generated by CMake, and its documentation is built with a combination of xsltproc and Doxygen. As dependencies go that’s not too bad, but the problem starts because the codebase lay dormant for a couple of years, and somewhere between three and six years of net development and build tooling efforts have passed it by.

There was no Dockerfile for the project, so I wrote one to see if I could get it to build and could understand all of the build dependencies. That took a dozen tries, but eventually something went through to completion.

The first thing I noted was that it not only depended on CMake but also a particularly relatively recent version of CMake. Ditto for Doxygen - not any elderly Doxygen will do, but only a relatively recent build. Doxygen has its own dependencies, lots of them - about a gigabyte of code to download and install.

Next was to pick up on a previous effort to build the whole thing under Travis CI. Now I have a relatively old build system that doesn’t have these old dependencies so I have to drag them in, and not only drag in some version but a particularly new version.

Dependencies satisfied, it’s time to build. I want this to work on my Raspberry Pi, so I build it there, and discover that the version of Doxygen there is too old (oops) so it doesn’t build the docs at all, never mind the cheery message from the build script that claims “TidyLib API documentation has been built”.

Next, to test. Does it pass all of its tests? Well, there is a test directory with a bunch of test cases, but the documentation says things like

Of course some of the tests were to say avoid a segfault found. Other tests were to visually compare the original input test file in a browser, with how the new output displayed in a browser. This is a purely VISUAL compare, and can not be done in code.

Frankly, a test that can’t be done in code is one that I can’t run in finite time. I build all of the tests, diff them with the reference directory, and hope that the differences (there are a few) aren’t fatal.

You see where this is going…

Frankly, it’s hard to get all of the tooling right. Travis CI has all of its own quirks, and you have to set up a build correctly to get correct build completion results. Once the build happens, you want some sort of test harness to take all of these test cases and turn them into red lights and green lights on your screen, and I don’t have that yet. Building binaries is yet another task - one project contributor put together a Jenkins install to take on that task. And so forth.

The piece of this that I care about is twofold. tidy as it stands now is frozen in 2009, and that’s before HTML5. tidy-html5 is caught in flux, with a successful project rescue to get it to build at all, but not yet the full-on automation and test infrastructure that projects like docker or node.js have that allow for ferocious parallel development with reasonable test and integration coverage.

The tidy project is very old (20 yrs), and if it’s going to pass a full suite of automated tests on every check-in for every platform, that’s going to take some consolidated effort - both by people who care about the code (but not so much how it’s packaged) and by people who care about the packaging (so that they can use the code). Not only does the code need to evolve, but the build tools around the code need to move forward.

Writing up your tweets

I use Twitter regularly, and will frequently post short things there that reflect some 140 character slice of what I’m thinking about. That’s very quick, but at some level deeply unsatisfying, because how much can you put into 140 characters? Never quite enough.

The challenge is to write up your tweets into something longer. Take the quip and turn it into a lead paragraph, then expand the lead paragraph into three or five or fifteen or fifty more. Explore at leisure, including excursions into thinking that don’t lend themselves to quippiness.

In retrospect some tweets would have better been unwritten. The process of twittering out a stream of observations gets in the way of assembling the longer context. We focus on the stream of things coming our way, never taking time to take stock of the situation and make it make more sense.

Bluetooth and bluez inside Docker

xml2json as part of a web parsing pipeline

The problem, simply put. You have a messy, real world HTML page; you want to parse it to pull out some key element of it, to pass along to some other task; you want that parsing pipeline to be as compact as possible, because you’re running on a small machine, and you’re doing the task frequently, and because the page may change out from under you at any time.

Clearly this is not an optimal task to have, but you live with the web you have, not with the web you want to have.

My approach to this task is to assemble a pipeline of Unix tools that are as simple as possible, and that each do one thing pretty well, and that in combination are all well-refined enough that there are few surprises.

curl is the workhorse for fetching pages. The invocation curl -m 15 -s http://example.com pulls that page from the net and feeds it to standard output, but times out after 15 seconds so your pipeline doesn’t completely fail if a remote site is down.

tidy is the first thing I look at when doing data transformation. tidy -q -asxml 2>/dev/null takes HTML and converts it, quietly and uncomplainingly, into XML. It’s predictable and pretty fast. tidy has been around since the dawn of the web, and it newly has a Github project and a shiny web page and a support consortium, so if you want to build from source it’s readily possible.

Given XML, convert it to JSON. Here there are quite a few choices, depending on which language and which parser you want to start with, and the dependency tree is deep. I am least satisfied with my alternatives here, thus the motivation for this writeup. In alphabetical order by language:

There is no POSIX standard xml2json command written in C, alas.

In C++ there’s an xml2json command and library from Cheedoong Ch’ng. The Github project says

xml2json is the first carefully written C++ library that converts XML document to JSON format. It’s already been used in the soft subtitle cross-domain solution at the server-end of Tencent Video (http://v.qq.com) and its CDNs.

In Dart, see Steve Hamblett’s xml2json.dart library. The Github project includes a set of unit tests, as well as explicit support for three conventions (Parker, Badgerfish, and Google Data) for doing the conversion. The test suites are welcomed.

In Go, see Darren Elwood (textnode)’s xml2json.go. The Github project includes sample code to parse both RSS and generic XML files.

Transform a stream of XML into a stream of JSON, without requiring a schema or structs, written in Go (golang.org)

In Node.JS, see node-xml2json from BugLabs which converts XML to JSON using node-expat. You can install it with npm install xml2json.

In Perl, see Isidro Vila Verde’s xml2json.pl, which has a Github project and is based on XSLT. “You may need to install some perl modules before using it” is the extent of the install instructions.

In Python, see the xmlutils package from Kailash Nadh and Yigal Lazarev. You can install it with pip install xmlutils, or look at the Github project.

xmlutils.py is a set of Python utilities for processing xml files serially for converting them to various formats (SQL, CSV, JSON). The scripts use ElementTree.iterparse() to iterate through nodes in an XML document, thus not needing to load the entire DOM into memory. The scripts can be used to churn through large XML files (albeit taking long :P) without memory hiccups.

Once the original HTML document emerges nicely formatted as JSON, it’s relatively easy to pick out elements from it in most cases. Two tools to do this are jq and jp.

jq, from Stephen Dolan, is a command-line JSON processor. Now at version 1.5, it has a web site and a Github repository, plus a handy online test site that lets you do interactive tests. It’s written in portable C.

jq is like sed for JSON data - you can use it to slice and filter and map and transform structured data with the same ease that sed, awk, grep and friends let you play with text.

jp, from James Saryerwinnie, is a command line version of the JMESPath query language for JSON, documented at http://jmespath.org/ . This is the same query language embedded into the Amazon Web Services command line (AWS CLI), and it has powerful and compact operators for extracting elements from a JSON document. There are libraries in Python, PHP, Javascript, Ruby, Lua, and Go that implement JMESPath. jp is written in Go, and has cross-development tools available.

So that’s the environment. It’s a bit of a pain to get all of those tools running on a new bare-metal machine, so my thought is to put everything into one Docker build and make it straightforward to wrap everything together. Ideally this will be a fairly minimalist build, and thus the challenge is to find the set of dependencies that is small enough.

Managing a Raspberry Pi OpenVPN with Salt

When you successfully install something complex, and then you fail to document one small thing about how you did it, the inclination is to over-document once you get it working again; thus, this expository account of managing a set of Raspberry Pi devices using the “salt” system. It’s not a complete start to finish writeup, just a pointer to the bits that I had forgotten.

This is mostly experimental at this point, and the collection of devices is relatively small. I am pretty convinced that the Pi 2 has plenty of gas to do more things than most people will try to do, but the original Pi is slow enough and limited enough to perhaps make this scheme impractical.

When it works, you have a set of Pi’s in various locations, all networked together into an OpenVPN configuration.

First to read is Virtually secure with openvpn, pillars, and salt which goes over the structure and motivation and design for an OpenVPN network that answers this problem:

How do I use salt to create and install the openvpn and client specific config files for each minion –on demand?

If you follow Dan Garthwaite’s design you get a reasonable approach - when you bring up a new client you get a new key generated, and on the server you run salt-key -a to accept the new one.

This leaves only the problem of bootstrapping on the Pi’s themselves. The version of ‘salt-minion’ that’s available from the default Debian system is very ancient, so that’s useless. What you want can be found at Setup a Debian Saltstack minion with a single command on the Server Network Tech blog; it brings in a new Debian source list from Salt so that the bootstrap is easier.

(I realize this isn’t a start to finish tutorial, just a few points of clarification for bringing up new nodes once you have the whole Salt environment running; sorry about that.)

The one thing I have left to do to make the setup more airtight is to automate the process of locking down the sshd_config so that “root” and “pi” can’t login from far away, so that I can keep their default passwords but restrict access to console access only.

Not mentioned here is that these Pi configuraiton are all on top of the Hypriot setup so that they can (and do) run Docker. Though Docker on the original Pi with salt-minion running doesn’t leave a ton of space spare for actual work to be done.

4 September 2015

Deep web

I think this whole “deep web” thing is mis-stated for the most part.

Since forever there have been Internet-based services that have been unreachable to the public unless you had some kind of specialized access. Private internal networks are the norm at companies of any size. Calling that the “hidden web”, “deep web” is non-sensical in that it’s not really hidden to anyone who has proper access to it - it’s just that you (and Google) don’t and won’t have that access.

Furthermore there are enormous numbers of home networks behind NAT devices where there are home servers that are either accidentally or deliberately off-limits. If you want to get that data onto the public network you actually have to do some work to make it happen.

I even have a batch of test services running on my laptop that don’t have any public presence at all beyond my keyboard.

There happen to be parts of the “secret web”, “deep web” etc where there’s some naughty content, but for the most part the Internet architecture makes it completely normal to have hosts that only have partial routing to the world, and most of the time that’s completely innocuous.

(As originally posted to Quora to answer the question, “How is the Deep Web so huge?)

Over the top networks, a history of building new systems on the wreckage of the old

This is a copy of a proposal for a talk at the Radical/Networks conference. The original proposal is on Github. The event runs October 24th and 25th (Saturday + Sunday) from 10a - 7p at NYU Poly in Brooklyn, NY.

Title: Over the top networks, a history of building new systems on the wreckage of the old

New networks often feed off of old ones, in ways that characteristically draw energy from aspects of those networks that are easy to take over and hard to defend. In this talk, I’ll look at the waves of creative destruction that are unleashed when network developers find existing infrastructure that can be exploited for new ends, and the ways that commoditized networks fight back to avoid being turned into dumb pipes.

The talk will look at the history and the future of these overlay or over-the-top networks, going back three decades for stories of networks like Usenet, electronic mail, and payments networks and how they started out parasitizing existing older networks only be overtopped by other interests as file sharing, security, and identity layers of newer networks. I’ll look at rules for developers of radical advances in networks and guidelines to avoid the pitfalls of unexpected dependencies and hidden traps.

Edward Vielmetti has been working on the Internet since 1985 from Ann Arbor, Michigan. His background includes work on the early commercialization and privatization of the Internet. Previous conference presentations include “WYSSA means all my love, darling: A social history of the Internet from the carrier pigeon to Antarctic morse code” (UPA 2006) and “Perils and Pitfalls of Practical Cybercommerce” (ARABANK 1996, Dubai, United Arab Emirates).

Terabytes, at least in theory

I have a new terabyte drive. It’s impossibly small, impossibly cheap, and just fast enough to accomplish astounding backup tasks in what seems like not enough time.

It is, however, a mass storage device that depends on rotating magnetic media, and as such it’s obsolete.

The drive cost less than $50. For about 10 times that right now, I can buy another terabyte drive that is solid state rather than spinning oxide storage. The bet that I have is that in three years, the cost of a terabyte of SSD drive will have come down so far - perhaps to $100 - that no one will have any interest in selling me a 1T drive that isn’t SSD.

My frame of reference for “what does har drive storage cost” dates back as far as the 10 megabyte, $1000 drive, which I really wanted and which I never bought for my own self. And then back a little bit further, when a 5 megabyte Davong drive was the bee’s knees.

What do you do with a terabyte? Lots of backups.

26 August 2015

RRDTool v1.5.4 on Raspberry Pi

The second in a series of efforts to get a suitable database for the task.

The challenge is that there’s a lot of data out there that changes all the time, and you want to gather it, make sure it’s sane, squirrel it away for later analysis, put together a pretty chart so that you can spot check it, feed it into other systems not yet fully dreamed of to cause actions, and so on. It’s the “big data” problem in that you have sensors that throw off a lot of data exhaust, and it’s the “internet of things” problem because mostly you’re looking at things with data and not say imagery or lots of freeform text.

Phase one of this effort was wrestling MongoDB into service, which ended up being only a handful of lines of code; but the concern expressed was the Mongo though fun and easy for small projects gets messy for big ones, and there’s good reason not to succumb to its siren song. The other reason to stop using Mongo when I did was that it really doesn’t love the Raspberry Pi architecture. Since there were only really a very few lines of code, and since that code works on my Mac, no time lost - but time to move on.

Phase two is a look at RRDtool, a “round robin” database very often used for network management efforts. One of the nice things about a round robin database is that the database itself is at its maximum size when you create it, and it doesn’t grow - rather, it cycles out the old data at whatever time resolution you want to get rid of it. With that in mind you can have a single database with 10 years worth of daily averages and a whole year of minute-by-minute data and it hardly takes up any room at all - perfect for the sort of “data logger” applications that I find myself in the middle of.

If you do apt-get install rrdtool on the Pi you get a working copy of RRDtool 1.4.7, compiled in 2012. That’s pretty long in the tooth, so I decided to pull down v1.5.4 from the RRDtool download area. The release notes note dependencies as follows

apt-get install libpango1.0-dev libxml2-dev

and once you have those pulled down the build is not fast but not difficult.

Pushing data into RRDtool is as simple as a shell script run out of cron. The first data point I’m graphing is “total spaces free in the Ann Arbor DDA parking garages”, and I’m collecting that on two machines - my Mac when it’s running, and the Pi at all times. A second graph I want to capture is some really basic weather information, and a third is inside-the-machine environmental sensing starting with the system temperature of the Pi.

It was really useful to make the first round of this be Mongo, because with Mongo you really don’t have to decide anything when you set things up - just dump in raw JSON documents and hope for the best.

First pass at MongoDB

Somehow I never learned SQL along the way. At various times I have learned enough of the database lingo (inner join, outer join etc) to know where I missing things, and I have a lot of experience with common Unix tools (grep, sed, awk) and some newfangled tools (jq, csvkit) to be able to turn a big pile of data into a smaller pile of data. But I don’t have database chops per se right now, and sometimes that’s a limiting factor on projects when the data sets are too big.

JSON formatted data seems to have won the data serialization wars, so that at least transforms the problem on input; with jq (now in release 1.5) you can take a stream of JSON formatted data that’s almost what you want and mangle it until it’s really what you want. Lots of web APIs spit out some flavor of JSON, and even if it’s not precisely formatted to be ready for input into something else, at least there’s a straightforward way to munge it into something better.

That leaves the task of dumping these JSON blobs into a database, in such a way that you can relatively easily get something back out of the database that answers what you are really trying to answer again. Here the world splits into two pieces: the SQL databases that have been taught how to deal with JSON (specifically PostgreSQL, as of version 9.3), and the so-called NoSQL databases (specifically MongoDB).

Even Mongo knows that scaling MongoDB is hard, though they frame it as the first 100 gigabytes of data stored in the system is easy. Fortunately, the first set of tasks I have to throw at this particular problem space are relatively small, databases measured in the 10s of megabytes. Pretty much anything will work - even grep - on this size data. The challenge becomes more of a how fast can you get something real running that has relatively simple queries, relatively performant results, and that can run on inexpensive hardware.

So I learned just enough Mongo to handle one tiny task: connect to an API, download some data periodically, clean it up enough so that it’s regular, and store it away; and then construct queries that are conceptually simple but that answer real questions. From there, do the ad hoc queries that prove to yourself that you have the right data, and then bootstrap some web app on top of all that to show off what you’ve done.

The results here accomplish a relatively complex task with a handful of lines of code. An API from a local agency is queried that has real time data. That data gets cleaned up and standardized and then inserted into a Mongo database. It’s all of 6 lines of code.

Listening to Radio Havana Cuba via a shortwave software defined radio

I’m listening to Radio Havana Cuba via a shortwave listening post set up with a software defined radio. The system, Great Lakes Listening Post, has four separate radios covering the 80, 40, 30, and 20 meter bands, all fed into the WebSDR software from PA3FWM. It’s located in Michigan, and thus approximates the radio reception I’d get from my own receiver.

R Havana Cuba is a good station to listen to. There’s interesting political commentary, fun music, and the occasional show about shortwave listening. The signal is usually strong and the frequency (6000 kHz) is easy to remember. Most international braodcasters have refocused their attention on Internet based broadcasts, but R Havana Cuba continues on with its shortwave service. My Sony ICF-2010 set from the 1980s still has a Cold War era present to that frequency.

The biggest advantage to web-based software radio is the ability to pick up stations from listening posts around the world. The WebSDR platform has stations with good antennas around the world, and they each allow multiple listeners to tune in to different stations without any of them monopolizing the receiver. Most web-based SDR systems focus on shortwave and HF amateur bands, but every so often you find a system that can tune domestic medium wave (AM) and long wave bands, which gives you a chance to hear programs that were never anticipated to be listened to from half way around the world.

23 August 2015

GQRX based radio scanner with gqrx-scan

gqrx-scan is a Perl script that drives the remote control of the gqrx software defined radio receiver to turn it into a scanner and automated recording tool.

The GQRX remote control protocol is well documented and fairly simple, which means that the bulk of the Perl script has to do with setting and getting options. You can drive your scanner from the GQRX configuration file, or you can monitor a specific band or range of frequencies. It will even toggle the record mode in GQRX so that you can save what you’re listening to a file.

Because the system is driven from a file, I would like to see more example files in the distribution - e.g. one that had all of the GMRS/FRS frequencies already typed in so that you could start scanning those bands immediately. But that’s a quibble, and more to the point that’s something that I can do next when the next time I’m ready to contribute to the project.

Credit goes to Github user khaytsus.

Using Github Issues and Waffle.io as a personal kanban board

I’ve been using Github issues to track my personal todo list for quite some time now, and since November I’ve been using Waffle.io to help that process.

Waffle in this use is a view of a series of queues in my todo list. Most everything that I’m not doing now lives in the “backlog” queue, which I try to keep to less than 100 just to keep myself sane. (At more than 100 there’s usually a lot of duplicates; at less than 100 I’m probably not acknowledging that there’s something to be done). The “Ready” and “In Progress” queues hold no more than 5 tasks apiece, since I want to keep everything in that state visible on the screen. Finally the “Done” queue reflects how much work there is that actually got accomplished over the last week, and I’d love to say that was hovering at about 50 tasks a week though right now it’s only at 34.

Github issues are more than capable of handling a personal todo list - indeed it has a lot of features that if you tried to use all of them you’d quickly find that too much of your time was perfecting your lists and not enough time was spent doing things. So the challenge is coming up with a simple configuration of the system that doesn’t ignore what might be done, but that focuses on what really needs to be done.

The inspriation for all of this is Jim Benson’s “Personal Kanban”, and his observation that really the task simplifies down to two rules:

  • Limit work in progress
  • Visualize your work

The rest is merely commentary.

(Now I get to drag task 3568, “Write up use of Waffle for Vacuum blog”, into the “Done” category, which is kinetically very satisfactory.)

At the corner of Ashley and Madison in Ann Arbor is the Washtenaw Dairy.

Ann Arbor City Council Preview for 17 August 2015

Ann Arbor City Council meets at 7:00 p.m. on Monday, 17 August 2015. A preview of the agenda is on Arborwiki, complete with a map, on the tag 2015-08-17CCMTG.

Some highlights to note:

The consent agenda includes $36,412 for the purchase of 21 Taser weapons. This is a sole-source purchase, to replace weapons purchased in 2003.

Also on the consent agenda is $83,000 for a sole-source purchase to renew the CityWorks license, which is used for asset tracking and maintenance of city owned resources. Of that amount, $20,000 will be billed back to the Washtenaw County Water Resources Commissioner.

$110,000 would go for the continuation of CLEMIS Public Safety Information Sharing Services for the Ann Arbor Police Department for FY2016. CLEMIS runs the dispatch system for police and allows “information sharing” with other departments. It’s run by Oakland County.

HathiTrust has digitized many decades of Ann Arbor City Council minutes, and it it proposed that the city approve publication of those minutes through HathiTrust by clearing the copyright on those materials.

Public hearings include one on deer management, and there’s a proposal on the agenda to spend $90,000 on some combination of a deer cull by sharpshooters and sterlization of deer. Expect this to be a long public hearing.

In development news, there’s a proposed condominium development on N Main St adjacent to North Main Park, and hearings on proposed annexations on Nixon Road and on Ann Arbor - Saline Road.

If you’re following the footing drain disconnect program closely, you’ll note that there’s a proposal to lend the builder of 618 South Main footing drain disconnect credits to allow for full occupancy of that building.

More of course at the full agenda. See you on the site plan barricades (er, see you at council).
