Oblique strategies as a service

Mar 24, 2017 #FaaS

In 1975, Brian Eno and Peter Schmidt collaborated on a card deck of strategies for overcoming creative blocks. This was produced in limited and somewhat varying editions, and when it escaped into the computing world at slightly later date, various electronic versions were produced (including one in FORTRAN). Prentiss Riddle noted some of this history in 2004.

The ‘oblique’ code is simple; a file with strategies, one per line, and a function to return a random line from that file. There are lots of random line functions out there, including the GNU “shuf” command, and various implementations done purely in the Unix shell.

In the tradition of exercising the card “Use an old idea” I present to you “Oblique strategies as a service”, implemented on Alex Ellis’s “Functions as a service” platform. The short version of how to do this involves starting from the Github repository and installing it according to the blog post at http://blog.alexellis.io/functions-as-a-service/ . Bring up the sample according to directions, then follow the discussion at https://github.com/alexellis/faas/issues/26 to see how to add a new function at runtime. At some point it boils down to

% curl localhost:8080/system/functions -d '
{"service": "oblique", "image": "vielmetti/faas-oblique", "envProcess": "/usr/bin/oblique", "network": "func_functions"}'

to instantiate the function, and then

% curl -d "" localhost:8080/functions/oblique

to call it.

Alex’s system relies on a long-running process in each containers, a so-called “watchdog” process that accepts requests, handles input/output to the function through the Go os/exec syscall, and marshalls requests. The actual function can be really anything including any command that communicates through stdio.

Some related work:

  • Amazon’s “Lambda” service
  • Glider Lab’s “cmd.io”, “commands as a service”; see announcing cmd.io, 2016 build on top of ssh
  • Ben Firshman’s “Funker”, Functions as Docker containers; see Funker repository

I have this running successfully on my Mac, in Docker 17.03.1-ce-rc1-mac3 (15924) on the “edge” release channel. Performance is on par with a local invocation of the same command.

Next steps? Why of course, port Functions as a Service to ARMv8. It’s already in the midst of being set up for ARM-based Raspberry Pi systems, so the portability work has been isolated. Follow the issue at https://github.com/alexellis/faas/issues/27 to see where we are going with that.
