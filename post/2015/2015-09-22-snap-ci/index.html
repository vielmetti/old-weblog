Snap CI

Snap CI is a cloud-based continuous integration system. Like Travis CI, which I have written about here, it lives in the cloud, lets you run programs triggered by GitHub commits, and offers a testing environment for free for open source projects which lets you familiarize yourself with the system before deciding to pay for it for your non-open-source efforts.

I’m looking at all of these tools now because this weblog is now managed through a process that involves checking things into Github, which makes it completely suitable for automation for common tasks every time I save a file. (Spell checking, link checking, really any kind of quality control you can imagine reducing to a tiny shell script.)

Like Travis CI, Snap CI lets you automate the build process. Where it diverges (and where it’s interesting) is that the build process is only the first part of a multi-stage pipeline that includes deployment at the end of the task only when a set of human decisions have been made as to whether the thing you made is ready to be shared.

This Snap CI blog post on why Travis CI and Snap CI are not the same thing is from Snap CI and it’s a useful overview.

The downside of multi-stage pipelines for deployment is that they are very hard to automate correctly, and very hard to demonstrate correct behavior with open source projects that generally struggle to have any automated test facilities at all. So while Travis CI is a very good fit for open source components (and has attracted a whole sub-industry of writing .travis.yml files to support those), it really doesn’t have the whole “continuous deployment” pattern sorted out. And while Snap CI looks like it would be great for saying “once we’re happy with this, push it out to the world”, that setup is not easy to pick up simply by inspecting other projects that have done it.
