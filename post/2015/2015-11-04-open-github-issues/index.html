Open Github issues and open source projects

Nov 4, 2015

As a first approximation, the number of open issues on an open source project that uses Github to track its issues is a metric of software complexity and project maturity.

It’s always worrisome to look at a project that has very few issues, because that’s usually a signal that no one else is using the code. Every bit of software has incomplete features that prompt a request, and bugs looking for a fix. No open issues at all suggest a dead project, or at least one that’s not actively attracting development.

Too many issues, on the other hand, suggest a system that’s so complex that no one person has a complete understanding of how the project works. The number of issues can grow beyond the point where a person could read and understand all the incoming bug reports and feature requests.

Every successful open source project deals with the struggle between adding new features to fill out the original vision of the designers and fixing enough bugs to make the system stable enough that it can be used in production without undue fears. This tug-of-war between features and bug fixes plays itself out over time, and sometimes difficult and complex bugs that only affect a few users get deprioritized over shiny new features that will attract a lot of new people to the project.

How many open issues can one person track? I’ll estimate that the number is about 100, plus or minus. So when a two person project has 50 open issues, it seems like it’s under control. A project that has over 1000 open issues (e.g. docker) can only raise some warning flags - how did it get to be so big? What sorts of issues can stick around for years and not get resolved? How on earth does one profess to have any overall understanding of a system that complex, short of devoting themselves to it?
